# 통신사 고객 이탈 예측 및 자동 리텐션 에이전트 프로젝트 계획서 (필수+선택)

> Telco Customer Churn Prediction & Automated Retention Agent
> 약 2개월(8주) 실전 모의 프로젝트 / 머신러닝·딥러닝·생성형 AI·LLM 엔지니어링·모델 서빙·AI Agent 통합
> 데이터셋: Kaggle Telco Customer Churn (`blastchar/telco-customer-churn`)
>
> v2 변경점: 분석·모델링 프로세스를 **EDA → Feature Engineering → Modeling(샘플링·다중 분류모형) → Ensemble** 흐름으로 구체화하고, **기본 Data / 부가 Data** 구분을 명시.

## 1. 프로젝트 개요

### 1) 프로젝트 주제명

**통신사 고객 이탈(Churn) 예측 모델 기반 자동 리텐션(고객 유지) 에이전트 개발**

한 문장으로 요약하면 다음과 같습니다.

> "떠날 것 같은 고객을 미리 찾아내고(예측), 그 고객에게 맞는 메시지와 혜택을 자동으로 제안하는(처방) AI 시스템을 만든다."

이 프로젝트의 핵심은 **예측에서 멈추지 않고 행동(액션)까지 자동화**한다는 점입니다. 단순 분류 모델 과제와 차별화되는 지점이며, "그래서 이 모델로 무엇을 했는가?"라는 면접 질문에 명확히 답할 수 있는 구조입니다.

### 2) 프로젝트 수행 배경

**(가) 비즈니스 배경**

- 통신 시장은 포화 상태라 신규 고객을 늘리기 어렵습니다. 따라서 **기존 고객을 지키는 일(Retention)**이 매출 방어의 핵심입니다.
- 일반적으로 신규 고객 한 명을 데려오는 비용이 기존 고객을 유지하는 비용보다 훨씬 큽니다. 이탈을 조금만 줄여도 수익에 큰 영향을 줍니다.
- 문제는 이탈을 **사후에** 알면 늦다는 점입니다. 고객이 해지한 뒤에는 되돌리기 어렵습니다.

**(나) 문제 정의 (Problem)**

1. 어떤 고객이 곧 이탈할지 **사전에** 알 수 없다.
2. 안다고 해도, **누구에게 / 어떤 메시지로 / 어떤 혜택을** 제안할지 사람이 일일이 판단하기 어렵다.
3. 대상 고객이 많아지면 **수작업으로는 감당이 불가능**하다.

**(다) 가치 정의 (Value)**

| 단계 | 질문 | 담당 기술 |
|---|---|---|
| 예측 | 누가 떠날까? | 머신러닝 / 딥러닝 |
| 해석 | 왜 떠날까? | 설명가능 AI(SHAP) |
| 처방 | 어떻게 잡을까? | 생성형 AI(LLM) |
| 실행 | 자동으로 무엇을 할까? | AI Agent |
| 운영 | 잘 되고 있나? | 모델 서빙 & 대시보드 |

즉 **"예측(Prediction)"과 "처방(Prescription)"을 하나의 파이프라인으로 연결**하는 것이 이 프로젝트의 가치입니다.

**(라) 학습 측면 배경**

수강한 전 과목(ML, DL, 생성형 AI/LLM, LLM 엔지니어링, 모델 서빙·대시보드, AI Agent)을 **하나의 흐름 안에서** 사용하므로, 개별 기술이 실제 제품에서 어떻게 맞물리는지 체득할 수 있습니다.

### 3) 프로젝트 목표 및 수행 내용

**(가) 목표 (측정 가능하게)**

정량 목표(예시 기준이며 데이터 분석 후 조정)

- 이탈 예측 모델 ROC-AUC 0.83 이상, PR-AUC 0.63 이상
- 이탈 고객 재현율(Recall) 0.75 이상 (떠날 고객을 놓치지 않는 것이 더 중요)
- 위험 고객 1건당 맞춤 리텐션 메시지 자동 생성 100% 자동화
- 위험도 구간별 액션 자동 분기 규칙 구현

정성 목표

- API 키 없이도 전체 데모가 동작(시뮬레이션 모드)
- 핵심 로직은 테스트(TDD)로 검증
- 비개발자도 이해할 수 있는 대시보드 제공

**(나) 수행 내용 (모듈 단위)**

| 모듈 | 명칭 | 핵심 수행 내용 |
|---|---|---|
| M1 | EDA(데이터 이해) | 기본정보 탐색, 결측치 분포 확인, 변수별 정보 확인 |
| M2 | Feature Engineering | 기본 Data 전처리, 파생변수 생성, 최종 데이터프레임 구성 |
| M3 | Modeling | 데이터 샘플링(Train/Test/Val·SMOTE), 다중 분류모형 생성·평가 |
| M4 | Ensemble | 상위 모델 결합(Voting/Stacking)으로 성능·안정성 향상 |
| M5 | 이탈 사유 해석 (권장·스트레치) | SHAP로 "왜 위험한지" 설명 — **고위험 고객에 한정** 적용, MVP에서는 선택 |
| M6 | 리텐션 메시지 생성 | LLM으로 고객 맞춤 메시지 자동 작성 |
| M7 | 리텐션 에이전트 | 위험도에 따라 액션 결정(유지 제안/상담 연결/관찰) |
| M8 | 대시보드 (+서빙 선택) | 모니터링 화면(권장) + 추론 API(선택·스트레치) |
| (공통) | 평가 & 테스트(TDD) | 단위 테스트, 성능 검증, 회귀 방지 |

**(다) 구현 우선순위 (필수 / 권장 / 선택·스트레치)**

모든 모듈·세부 항목을 우선순위로 분류했습니다. 일정이 빠듯하면 **필수 → 권장 → 선택** 순으로 진행하면 됩니다.

| 단계/항목 | 우선순위 | 비고 |
|---|---|---|
| M1 EDA | **필수** | 모든 분석의 출발점 |
| M2 전처리·최종 데이터프레임 | **필수** | 모델 입력 생성 |
| M2 파생변수 생성 | 권장 | 성능·해석력 향상 |
| M3 Train/Test/Val 분리 | **필수** | 올바른 평가의 전제 |
| M3 SMOTE 오버샘플링 | 권장 | 대안: 클래스 가중치(class_weight) |
| M3 RandomForest (베이스라인) | **필수** | 기준선 모형 1개 |
| M3 부스팅(XGBoost/LightGBM/CatBoost) | 핵심(권장) | 최소 1종, 권장 2~3종 |
| M3 MLPClassifier | 선택 | 딥러닝 비교군, 성능 보통·스케일링 민감 |
| M3 SVC | 선택 | 7천 행에서 학습 느림, 우선순위 낮음 |
| M4 Ensemble | 권장·스트레치 | 단일 최고 모형보다 개선될 때만 채택 |
| M5 SHAP 해석 | 권장·스트레치 | 고위험 고객 한정 |
| M6 LLM 메시지(시뮬레이션 포함) | **필수(핵심)** | 프로젝트 차별점 "처방" |
| M7 리텐션 에이전트(규칙 기반) | **필수(핵심)** | "행동 자동화" |
| M7 LLM 툴 호출 에이전트 | 선택·스트레치 | 에이전트 고도화 |
| M8 추론 API(모델 서빙) | 선택·스트레치 | 시간 여유 시. 없어도 대시보드·스크립트로 시연 가능 |
| M8 대시보드 | 권장 | 시연 가치 큼 |
| 공통 TDD | **필수** | 핵심 로직 보호 |

> **MVP 최소 구성(필수)**: EDA → 전처리 → (분할 + RandomForest + 부스팅 1종) → 시뮬레이션 메시지 → 규칙 기반 에이전트 → 발표.
> **여기에 권장 추가**: 파생변수, SMOTE, 부스팅 2~3종, 대시보드, 공정성 점검.
> **스트레치(시간 여유 시)**: Ensemble, SHAP 개인화 사유, LLM 툴 호출 에이전트, MLP·SVC 비교, **추론 API(모델 서빙)**.

### 5) 활용되는 데이터 출처 및 형태

**(가) 출처 및 다운로드**

- 데이터셋 페이지: https://www.kaggle.com/datasets/blastchar/telco-customer-churn
- 파일: `WA_Fn-UseC_-Telco-Customer-Churn.csv`
- 규모: 7,043행 × 21열, 단일 CSV 파일
- 라이선스: 학습/연구용 공개 데이터 (Kaggle 로그인 후 다운로드)
- 내려받기 방법: ① 데이터셋 페이지의 Download 버튼, ② `kagglehub` 패키지, ③ Kaggle API(CLI). API 사용 시 Kaggle 계정에서 발급한 토큰(`kaggle.json`)이 필요합니다.

**(나) 기본 Data / 부가 Data 구분**

단일 CSV이지만, 분석·설계를 명확히 하기 위해 21개 컬럼을 **기본 Data(필수 계정 정보)**와 **부가 Data(선택 부가 서비스)**로 나눕니다.

- **기본 Data (Core)** — 가입 시 반드시 존재하는 고객·계약·요금 정보. 이탈 예측의 1차 핵심 신호.
- **부가 Data (Add-on)** — 고객이 선택적으로 가입하는 부가 서비스 현황. 충성도·연계 정도를 나타내며 파생변수(가입 서비스 수)의 원천.

| 구분 | 그룹 | 컬럼 |
|---|---|---|
| **기본 Data** | 식별자 | customerID |
| **기본 Data** | 인구통계 | gender, SeniorCitizen, Partner, Dependents |
| **기본 Data** | 가입기간 | tenure |
| **기본 Data** | 계약·결제 | Contract, PaperlessBilling, PaymentMethod |
| **기본 Data** | 금액 | MonthlyCharges, TotalCharges |
| **기본 Data** | 타깃 | Churn |
| **부가 Data** | 통신 서비스 | PhoneService, MultipleLines, InternetService |
| **부가 Data** | 부가 서비스 | OnlineSecurity, OnlineBackup, DeviceProtection, TechSupport, StreamingTV, StreamingMovies |

> 활용 원칙: 기본 Data는 모든 모델의 **필수 입력**으로 사용하고, 부가 Data는 **그대로 입력 + 가입 서비스 수 등 파생변수**로 가공합니다. 두 그룹을 합쳐 최종 학습용 데이터프레임을 만듭니다.

| 서비스 | OnlineBackup | 범주 | 온라인 백업 |
| 서비스 | DeviceProtection | 범주 | 기기 보호 |
| 서비스 | TechSupport | 범주 | 기술 지원 |
| 서비스 | StreamingTV | 범주 | 스트리밍 TV |
| 서비스 | StreamingMovies | 범주 | 스트리밍 영화 |
| 계약/결제 | Contract | 범주 | 계약유형(Month-to-month/One year/Two year) |
| 계약/결제 | PaperlessBilling | 범주 | 전자청구 여부 |
| 계약/결제 | PaymentMethod | 범주 | 결제수단(전자수표/우편수표/은행이체/카드) |
| 금액 | MonthlyCharges | 실수 | 월 요금 |
| 금액 | TotalCharges | 실수 | 누적 요금 |
| 타깃 | Churn | 범주(Yes/No) | 이탈 여부(예측 대상) |

**(라) 데이터 사용 시 주의점 (학생들이 자주 막히는 부분)**

1. `TotalCharges`는 숫자처럼 보이지만 **공백 문자열 11건**이 섞여 있어 그대로 읽으면 문자열로 인식됩니다. 숫자로 변환 후 결측 처리가 필요합니다.
   - 참고: 이 11건은 모두 `tenure`(가입기간)가 0인 **신규 고객**입니다. 아직 누적 요금이 없어 공백인 것이므로, 0 또는 중앙값으로 대체하는 근거가 명확합니다.
2. `Churn`은 Yes/No 문자열이므로 1/0으로 변환해야 합니다.
3. 이탈 비율이 약 **26.5%**로 불균형합니다. 정확도(Accuracy)만 보면 안 되고 재현율·PR-AUC를 함께 봐야 합니다.
4. `SeniorCitizen`은 이미 0/1 정수이므로 추가 인코딩이 불필요합니다.

## 2. 상세 구현/분석 방법 및 예시

### 1) 구현/분석 프로세스

각 모듈을 "문제·가치정의 → 분석(EDA) → 설계(인터페이스/테스트 정의) → TDD 개발"의 순서로 진행하고, 다음 모듈로 넘어갑니다. 별도의 고정된 스프린트 반복 없이, 모듈(M1~M8) 단위로 순차 진행하며 필요한 경우에만 앞 단계로 되돌아가 보완합니다.

분석·모델링 핵심 흐름은 다음과 같습니다.

```
[EDA] ──▶ [Feature Engineering] ──▶ [Modeling] ──▶ [Ensemble]
  │             │                       │              │
기본정보·     기본Data 전처리·       Train/Test/Val·  상위 모델
결측·변수     파생변수·최종DF        SMOTE·다중모형    결합
```

전체 데이터 흐름(엔드투엔드)은 다음과 같습니다.

```
고객 데이터(CSV)
      │
      ▼
 [EDA / 전처리 / 피처]  ──▶  [모델링 + 앙상블]  ──▶  이탈 확률(0~1)
                                                      │
                                  ┌───────────────────┘
                                  ▼
                            [SHAP 사유 해석]  ──▶  "왜 위험한가" 근거
                                  │
                                  ▼
                          [리텐션 에이전트] ── 위험도 분기 ──┐
                                  │                          │
                      위험 높음   │   위험 중간   │  위험 낮음
                        ▼         ▼              ▼
                [LLM 맞춤 메시지] [상담 연결]   [관찰만]
                                  │
                                  ▼
                         [대시보드 / 로그 기록]
```

> 위 흐름에서 **[SHAP 사유 해석]은 권장·스트레치 단계**입니다. 생략 시 위험도만으로 액션을 결정하며, 적용 시 고위험 고객에 한정해 개인화 사유를 더합니다.

### 2) 구현/분석 방법론

**(가) 분석 방법론: CRISP-DM 매핑**

| CRISP-DM 단계 | 이 프로젝트에서 하는 일 |
|---|---|
| 비즈니스 이해 | 이탈의 비즈니스 비용 정의, 목표 지표 설정 |
| 데이터 이해 | EDA: 기본정보·결측·변수별 분포 파악 |
| 데이터 준비 | 전처리, 인코딩, 파생변수, 샘플링(SMOTE 포함) |
| 모델링 | 다중 분류모형 학습·비교, 앙상블, 임계값 튜닝 |
| 평가 | Recall/PR-AUC, 비즈니스 비용 기반 평가 |
| 배포 | API 서빙 + 에이전트 + 대시보드 |

**(나) 데이터 샘플링·불균형 처리 방법론**

- **데이터 분리**: Train(학습) / Test(모델 비교·평가) / Val(제출용 홀드아웃)로 분리하며, 분리 시 이탈 비율을 유지(stratify).
- **SMOTE 오버샘플링(권장 — 필수 아님)**: 소수 클래스(이탈) 보강을 위해 SMOTE를 적용하되, **반드시 학습(Train) 데이터에만** 적용합니다(Test·Val에 적용하면 데이터 누수·성능 착시 발생). 더 간단한 대안으로 **클래스 가중치(class_weight)**만 사용해도 됩니다.
- **보조 기법**: 클래스 가중치도 병행 실험하고, 기본 0.5가 아닌 비용 반영 임계값으로 조정합니다.

**(다) 분류모형 선정 및 평가 방법론**

서로 다른 계열의 모형을 비교해 데이터에 가장 적합한 모델을 찾습니다.

| 모형 | 계열 | 우선순위 | 특징 / 선정 이유 |
|---|---|---|---|
| RandomForestClassifier | 배깅 트리 앙상블 | **필수(베이스라인)** | 안정적 기준선, 과적합에 강건 |
| XGBoost | 부스팅 | 핵심(권장) | 불균형 보정·강력한 성능 |
| LightGBM | 부스팅 | 핵심(권장) | 빠르고 메모리 효율적 |
| CatBoost | 부스팅 | 핵심(권장) | 범주형 처리 우수, 적은 튜닝으로 양호 |
| MLPClassifier | 신경망(딥러닝 비교군) | 선택 | 비선형 패턴 학습, 스케일링 민감·성능 보통 |
| SVC | 서포트 벡터 머신 | 선택 | 마진 기반, 7천 행에서 학습 느림·우선순위 낮음 |

> 모형은 **RandomForest(베이스라인) + 부스팅 1~3종**을 먼저 확정한 뒤, 시간 여유가 있을 때 MLP·SVC를 비교군으로 추가하는 순서를 권장합니다.

- **평가 지표**: 이탈 고객(소수)을 놓치는 손해가 크므로 **Recall과 PR-AUC를 주지표**로, ROC-AUC·F1은 보조로 봅니다. 모든 모형을 동일한 Test 셋에서 같은 지표로 비교합니다.
- **교차검증**: 모형 비교의 신뢰성을 위해 계층적 교차검증(Stratified K-Fold)을 병행할 수 있습니다.

**(라) 앙상블(Ensemble) — 권장·스트레치(필수 아님)**

- 단일 모형으로도 프로젝트는 성립하며, 앙상블은 성능·안정성을 더 끌어올리고 싶을 때 진행합니다.
- 단일 모형 비교 후 **상위 성능 모형(예: 부스팅 3종 또는 계열이 다른 상위 모형)**을 결합합니다.
- **Soft Voting**(예측 확률 평균) 또는 **Stacking**(상위 모형의 출력을 메타 모형이 재학습)으로 일반화 성능과 안정성을 높입니다.
- 앙상블이 단일 최고 모형 대비 실제로 개선되는지 Test 셋에서 검증하고, 개선이 없으면 단일 모형을 채택합니다(과한 복잡도 지양).

**(마) 설명가능성(SHAP) — 권장(필수 아님)**

최종 채택 모델이 왜 그 고객을 위험하다고 봤는지 SHAP 값으로 설명하고, 이 사유를 LLM 메시지 생성의 입력으로 연결합니다(예측과 처방의 연결고리). 단, 예측→액션 흐름 자체는 SHAP 없이도 동작하므로 **필수 단계는 아니며 권장·스트레치**로 둡니다.

- **적용 범위 축소**: 전체 고객이 아니라 **액션 대상(고위험) 고객에만** SHAP을 적용해 계산 부담을 줄입니다.
- **경량 대체안**: 일정이 부족하면 모델 전역 피처 중요도(`feature_importances_`)나 상위 피처 규칙으로 대체할 수 있습니다(단, 고객별 개인화 사유는 약해짐).
- **MVP 분리**: MVP는 위험도 기반 액션까지 완성하고, SHAP 기반 개인화 사유는 그 위에 얹는 단계로 진행합니다.

**(바) LLM 프롬프트 설계 원칙**

- 입력: 고객 특징 + 이탈 사유(SHAP 상위 요인) + 위험도
- 출력: 정중하고 구체적인 리텐션 메시지(혜택 제안 포함)
- 가드레일: 과장/허위 혜택 금지, 사실 기반, 톤 일관

**(사) 에이전트 의사결정 규칙(초기 버전은 규칙 기반)**

| 위험도 구간 | 등급 | 액션 |
|---|---|---|
| 0.7 이상 | 고위험 | LLM 맞춤 메시지 생성 + 프리미엄 혜택 제안 |
| 0.4 ~ 0.7 | 중위험 | 상담사 연결 추천 + 표준 혜택 |
| 0.4 미만 | 저위험 | 별도 액션 없이 관찰 |

**(아) TDD 방법**

테스트를 먼저 정의하고(Red) → 통과하는 최소 구현(Green) → 정리(Refactor)합니다. 전처리, 샘플링, 임계값 분기, 에이전트 규칙처럼 **입출력이 명확한 부분**부터 테스트로 보호합니다.

**(자) 공정성/편향(Fairness) 점검**

- 민감 속성(예: `gender`, `SeniorCitizen`)별로 **재현율·정밀도·이탈 예측률 격차**를 측정합니다.
- 격차가 크면 임계값을 집단별로 점검하거나, 해당 속성을 입력에서 제외하는 실험을 합니다(제외해도 상관 피처로 정보가 누수될 수 있음을 함께 분석).

**(차) LLM 메시지 품질 평가**

| 평가 항목 | 확인 방법 |
|---|---|
| 가드레일 준수 | 금지어/허위 혜택 문구 미포함 자동 점검 |
| 사유 반영 | SHAP 상위 사유가 메시지에 반영되었는지 확인 |
| 톤/형식 | 정중체·길이 제한·필수 요소(혜택 제안) 포함 여부 |
| 사실성(선택) | LLM-as-a-judge로 사실 일치 여부 보조 채점 |

**(카) 에이전트 고도화 (스트레치 목표 — AI Agent 과목 연계)**

초기 버전은 규칙 기반((사))으로 빠르게 완성하되, 시간이 남으면 **LLM 기반 에이전트(툴 호출)**로 확장합니다. 위험도 조회·사유 조회·메시지 생성·상담 연결을 각각 툴(tool)로 정의하고, LLM이 호출 순서를 스스로 결정하게 합니다.

### 3) 구현/분석 상세 내용 및 예시

각 모듈의 목적·입력·처리·출력과 핵심 설계 포인트를 정리합니다. (수치는 설명용 예시이며 실제 분석 후 달라질 수 있습니다.)

#### [M1] EDA (데이터 이해)

사용자가 정의한 세 가지 관점으로 데이터를 탐색합니다.

| 단계 | 내용 |
|---|---|
| **기본정보 탐색** | 데이터 크기(7,043×21), 컬럼별 자료형, 기본 Data/부가 Data 구성, 타깃(Churn) 클래스 비율(약 26.5% 이탈) 확인 |
| **결측치 분포 확인** | 컬럼별 결측 현황 점검 — 특히 `TotalCharges` 공백 11건(모두 신규 고객)을 식별하고 처리 방향 결정 |
| **변수별 정보 확인** | 수치형(분포·이상치·왜도), 범주형(빈도), **각 변수와 이탈률의 관계**(예: 월단위 계약·짧은 tenure·광인터넷의 높은 이탈률) 파악 |

> 예시(목표 산출): 계약유형별 이탈률 비교, tenure 구간별 이탈률, 부가 서비스 가입 수와 이탈의 관계 등 인사이트 도출.

#### [M2] Feature Engineering

| 단계 | 내용 |
|---|---|
| **기본 Data 전처리** | `TotalCharges` 숫자 변환·결측 대체, `Churn` 1/0 변환, 수치형 표준화, 범주형 인코딩(원-핫) |
| **파생변수 만들기** | ① 가입 서비스 수(num_services): 부가 Data 합산 ② 평균 월 지출(avg_monthly_spend = TotalCharges/tenure) ③ tenure 구간화(신규/중기/장기) 등 |
| **최종 데이터프레임 만들기** | 기본 Data + 부가 Data + 파생변수를 결합해 모델 입력용 최종 DF 구성, `customerID` 등 식별자 제외 |

> 핵심 포인트: 전처리·인코딩은 학습 데이터로만 적합(fit)하고 검증/제출 데이터에는 적용(transform)만 하여 누수를 차단합니다.

#### [M3] Modeling

| 단계 | 내용 |
|---|---|
| **데이터 샘플링** | Train/Test/Val(제출용)로 분리(stratify), 학습 데이터에만 SMOTE 오버샘플링 적용 |
| **분류모형 생성·평가** | RandomForest, MLP, SVC, XGBoost, LightGBM, CatBoost 6종을 동일 조건에서 학습 |
| **모형 비교** | Test 셋에서 Recall·PR-AUC·ROC-AUC·F1로 비교표 작성, 임계값 튜닝 |
| **출력** | 모형별 성능 리포트, 최우수 모형 후보 선정 |

> 예시(목표 성능): 부스팅 계열에서 ROC-AUC 0.83+, PR-AUC 0.63+, Recall 0.75+ (임계값 조정 적용) 기대.

#### [M4] Ensemble — 권장·스트레치

| 항목 | 내용 |
|---|---|
| 필수 여부 | **필수 아님(권장)**. 단일 모형으로도 프로젝트 성립 |
| 목적 | 상위 모형을 결합해 일반화 성능·안정성 향상 |
| 방법 | Soft Voting(확률 평균) 또는 Stacking(메타 모형) |
| 검증 | 앙상블이 단일 최고 모형보다 실제로 개선되는지 Test 셋에서 확인, 개선 없으면 단일 모형 채택 |
| 출력 | 최종 채택 모델(파일로 저장) + 최종 성능 리포트 |

#### [M5] 이탈 사유 해석 (SHAP) — 권장·스트레치

| 항목 | 내용 |
|---|---|
| 필수 여부 | **필수 아님(권장)**. 예측→액션 흐름은 SHAP 없이도 동작 |
| 목적 | "왜 이 고객이 위험한가"를 사람이 읽을 수 있게 설명 |
| 적용 범위 | **고위험(액션 대상) 고객에 한정** 적용 |
| 처리 | 최종 모델에 SHAP 적용 → 상위 위험/보호 요인 추출 |
| 출력 | 위험/보호 요인 목록(기여도 포함) — LLM 입력으로 연계 |
| 대체안 | 일정 부족 시 전역 피처 중요도/상위 피처 규칙으로 대체 |

> 예시(개념): 한 고위험 고객(이탈 확률 0.82)의 위험 요인 — 월단위 계약, 짧은 가입기간, 광인터넷 / 보호 요인 — 기술지원 가입.

#### [M6] 리텐션 메시지 생성 (LLM, 시뮬레이션 모드 지원)

| 항목 | 내용 |
|---|---|
| 목적 | 고객·사유·위험도에 맞춘 정중한 리텐션 메시지 자동 작성 |
| 처리 | 프롬프트 구성 → LLM 호출(키 없으면 템플릿 기반 시뮬레이션으로 폴백) |
| 출력 | 맞춤 리텐션 메시지 텍스트 |
| 핵심 포인트 | API 키 없이도 전체 데모 동작, 가드레일(허위·과장 혜택 금지) 준수 |

#### [M7] 리텐션 에이전트 (의사결정)

| 항목 | 내용 |
|---|---|
| 목적 | 위험도에 따라 적절한 리텐션 액션을 자동 결정 |
| 처리 | 위험 구간 판정(고/중/저) → 액션 분기, 고위험은 메시지 생성 연계 |
| 출력 | 액션(메시지 발송 / 상담 연결 / 관찰) + (해당 시) 생성 메시지 |

> 예시(개념): 위험도 0.82 → 메시지 발송, 0.52 → 상담사 연결, 0.18 → 관찰.

#### [M8] 서빙 & 대시보드

- **추론 API**: 고객 정보를 받아 `{이탈확률, 사유, 액션, 메시지}`를 반환. 최종 저장 모델을 로드해 사용.
- **대시보드**: 전체 이탈 위험 분포, 고위험 고객 리스트, 액션 현황, 모델 성능 추이를 시각화.

```
[대시보드 화면 구성 예시]
┌───────────────────────────────────────────────┐
│ 전체 이탈 위험 분포   |  이번 주 고위험 고객 수 │
├───────────────────────────────────────────────┤
│ 고위험 고객 테이블 (ID / 위험도 / 사유 / 액션) │
├───────────────────────────────────────────────┤
│ 액션 현황(발송/상담연결/관찰)  | 모델 성능 추이 │
└───────────────────────────────────────────────┘
```

#### [공통] 평가 & 테스트(TDD)

- 입출력이 명확한 핵심 로직(데이터 로드·전처리·샘플링·임계값 분기·에이전트 규칙)을 자동화된 테스트로 보호합니다.
- 성능 임계값 회귀 테스트로 모델 품질 저하를 조기에 감지하고, 통합 단계에서는 엔드투엔드 회귀 테스트로 전체 흐름을 검증합니다.

### 4) 데이터 리소스 활용 상세

**(가) 컬럼별 활용 전략**

| 활용 목적 | 사용 컬럼(그룹) | 방법 |
|---|---|---|
| 예측 입력(필수) | 기본 Data: tenure, MonthlyCharges, Contract 등 | 인코딩/스케일링 후 모델 입력 |
| 예측 입력(보조) | 부가 Data: 인터넷·부가 서비스 | 인코딩 후 입력 |
| 파생 변수 | 부가 Data 전체 | 가입 서비스 개수(num_services) 합산 |
| 파생 변수 | tenure, TotalCharges | 평균 월 지출 = TotalCharges / tenure |
| 위험 해석 | 전체 피처 | SHAP 기여도 |
| 메시지 생성 | SHAP 상위 사유 + 계약/요금 | LLM 프롬프트 입력 |

**(나) 데이터 누수(leakage) 방지 원칙**

- 전처리(인코딩·스케일링·결측대체)와 **SMOTE는 반드시 학습 데이터로만** 수행하고, 검증/제출 데이터에는 변환만 적용합니다.
- 타깃(Churn)에서 파생된 정보를 입력 피처로 넣지 않습니다.
- `customerID`는 식별자이므로 학습 피처에서 제외합니다.

**(다) 파생 변수 설계**

- **가입 서비스 개수(num_services)**: 부가 Data 가입 수를 합산해 충성도·연계 정도를 표현.
- **평균 월 지출(avg_monthly_spend)**: 누적 요금을 가입기간으로 나눈 값. 신규 고객(tenure=0)은 분모 처리에 유의.
- **tenure 구간(group)**: 신규/중기/장기로 구간화해 비선형 패턴을 단순화.

**(라) 데이터 버전 관리**

- 원본 데이터는 절대 수정하지 않고, 전처리 결과만 별도 산출물로 저장합니다.
- 전처리·샘플링 규칙이 바뀌면 결과 파일명에 버전을 붙여 이력을 관리합니다.

### 5) 예상 계획 및 일정

> **⏱ 일정 현실성 검토 (결론)**
> 8주에 **모든 것**(6종 모델 + 앙상블 + SHAP + 서빙 API + 대시보드)을 하는 것은 4인 학생팀에게 **무리**입니다. 하지만 아래처럼 **필수 범위로 좁히면 8주에 충분히 완수 가능**합니다.
> - **필수 라인**(EDA → FE → 베이스라인+부스팅 → LLM 메시지 → 규칙 에이전트 → 발표)은 8주에 여유 있게 가능.
> - **권장**(파생변수·SMOTE·부스팅 2~3종·대시보드·공정성)은 진도가 맞으면 추가.
> - **스트레치**(앙상블·SHAP·LLM 에이전트·MLP/SVC·서빙 API)는 빠른 팀만.
> - **7주차에 통합·버퍼**를 두어 앞선 지연을 흡수합니다.

아래 일정은 **필수 우선**으로 구성했고, 각 주차에 필수/권장/선택을 함께 표시했습니다.

| 주차 | 단계 | 필수 (반드시) | 권장 / 선택·스트레치 | 주요 산출물 |
|---|---|---|---|---|
| 1주 | 셋업 + EDA | 환경 구성, 데이터 적재·검증, EDA(기본정보·결측·변수별) | — | 계획서 확정, EDA 결과 |
| 2주 | Feature Engineering | 기본 Data 전처리, 최종 데이터프레임 | (권장) 파생변수 | 전처리 파이프라인 |
| 3주 | 베이스라인 모델링 | 분할(Train/Test/Val), RandomForest, 평가체계(Recall/PR-AUC)·임계값 | (권장) SMOTE | 베이스라인 리포트 |
| 4주 | 모델 고도화·확정 | 부스팅 1~3종 비교, 임계값 튜닝, 최종 모델 확정 | (선택) MLP·SVC / (스트레치) Ensemble | 모델 비교표, 최종 모델 |
| 5주 | 처방① LLM 메시지 | LLM 메시지 생성(시뮬레이션 포함), 가드레일 | (스트레치) SHAP 개인화 사유 | 메시지 생성 모듈 |
| 6주 | 처방② 리텐션 에이전트 | 규칙 기반 에이전트(위험 분기), 액션 로깅 | (스트레치) LLM 툴 호출 에이전트 | 에이전트 모듈, 통합 추론 흐름 |
| 7주 | 통합 + 대시보드 + 버퍼 | 엔드투엔드 통합, 밀린 작업 따라잡기 | (권장) 대시보드·공정성 점검 / (선택) 추론 API 서빙 | 통합 데모 |
| 8주 | 마무리·발표 | E2E 회귀 테스트, 발표자료, README | (선택) 리팩토링 | 데모, 발표자료 |

> 각 주차의 **필수 항목에는 해당 핵심 로직 테스트(TDD)**를 함께 작성합니다(데이터 로드·전처리·임계값 분기·에이전트 규칙 등).

**(가) 역할 분담 예시 (4인 팀 가정)**

- 데이터/EDA·FE 담당: M1~M2
- 모델링 담당: M3~M4 (Ensemble은 스트레치)
- 해석/LLM·에이전트 담당: M5~M7
- 통합/대시보드 담당: M8 대시보드 + E2E 통합 (서빙 API는 선택, 테스트는 전원 공동)

**(나) 리스크 및 대응**

| 리스크 | 대응 |
|---|---|
| 불균형으로 성능이 낮게 보임 | Recall/PR-AUC 중심 평가, SMOTE·임계값 튜닝 |
| SMOTE 과적합/누수 | 학습 데이터에만 적용, Test·Val은 원본 분포 유지 |
| 모형이 너무 많아 일정 압박 | 베이스라인 먼저 확정 후 부스팅·앙상블 순으로 확장 |
| LLM API 비용/키 부재 | 시뮬레이션 모드로 전체 데모 동작 보장 |
| 일정 지연 | M1~M5(모델) 우선 완성, 서빙/대시보드는 최소기능(MVP)으로 |
| 설명가능성(SHAP) 구현 부담 | 필수 아님(권장)으로 처리 — 고위험 고객 한정 적용, 부족 시 전역 피처 중요도로 대체 |
| 데이터 누수 | 전처리·샘플링을 학습셋 한정으로 구조화 |
| 라이브러리 호환 오류(예: Python 3.14) | 3.11/3.12 가상환경 + 버전 고정으로 팀 환경 통일 |
| 모델 편향(특정 집단 불이익) | 집단별 지표 격차 측정·리포트(공정성 점검) |
| LLM 메시지 품질 편차 | 가드레일 자동 점검 + 평가 루브릭 적용 |
| 실험 재현 불가 | 난수 시드 고정, 실험 기록(파라미터·지표) 관리 |
