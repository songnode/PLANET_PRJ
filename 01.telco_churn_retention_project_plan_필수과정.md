## 0. 필수 범위 한눈에 보기

```
[필수 라인]
EDA → 전처리/최종DF → (분할 + RandomForest + 부스팅 1종 + 평가) → LLM 메시지(시뮬레이션) → 규칙 기반 에이전트 → 통합·발표
```

| 포함(필수) | 제외(이번 범위 아님) |
|---|---|
| EDA(기본정보·결측·변수별) | 다중 모형 6종 비교, MLP·SVC |
| 전처리 + 최종 데이터프레임 | SMOTE 오버샘플링 |
| 분할(Train/Test/Val) | 앙상블(Ensemble) |
| RandomForest + 부스팅 1종 | SHAP 사유 해석 |
| Recall/PR-AUC 평가·임계값 | 추론 API(모델 서빙) |
| LLM 리텐션 메시지(시뮬레이션) | 대시보드 |
| 규칙 기반 리텐션 에이전트 | LLM 툴 호출 에이전트 |
| 핵심 로직 TDD | 공정성 정량 점검 |

> 불균형 대응은 SMOTE 대신 **클래스 가중치(class_weight)**로 처리합니다(추가 라이브러리 불필요).

## 1. 프로젝트 개요

### 1) 프로젝트 주제명

**통신사 고객 이탈(Churn) 예측 모델 기반 자동 리텐션(고객 유지) 에이전트 개발**

> "떠날 것 같은 고객을 미리 찾아내고(예측), 그 고객에게 맞는 메시지를 자동으로 제안하는(처방) AI 시스템을 만든다."

핵심은 **예측에서 멈추지 않고 행동(액션)까지 자동화**한다는 점입니다. 단순 분류 모델 과제와 차별화되며, "그래서 이 모델로 무엇을 했는가?"에 명확히 답할 수 있습니다.

### 2) 프로젝트 수행 배경

**(가) 비즈니스 배경**

- 통신 시장은 포화 상태라 신규 고객 확보가 어렵습니다. 따라서 **기존 고객 유지(Retention)**가 매출 방어의 핵심입니다.
- 신규 고객 확보 비용이 기존 고객 유지 비용보다 훨씬 큽니다. 이탈을 조금만 줄여도 수익 영향이 큽니다.
- 이탈을 **사후에** 알면 늦습니다. 해지 전에 선제 대응해야 합니다.

**(나) 문제 정의 (Problem)**

1. 어떤 고객이 곧 이탈할지 **사전에** 알 수 없다.
2. 안다고 해도 **누구에게 / 어떤 메시지를** 보낼지 사람이 일일이 판단하기 어렵다.
3. 대상이 많아지면 **수작업으로는 감당 불가**하다.

**(다) 가치 정의 (Value)**

| 단계 | 질문 | 담당 기술 |
|---|---|---|
| 예측 | 누가 떠날까? | 머신러닝 |
| 처방 | 어떻게 잡을까? | 생성형 AI(LLM) |
| 실행 | 자동으로 무엇을 할까? | AI Agent(규칙 기반) |

즉 **"예측(Prediction)"과 "처방(Prescription)"을 하나의 파이프라인으로 연결**하는 것이 핵심 가치입니다.

### 3) 프로젝트 목표 및 수행 내용

**(가) 목표 (측정 가능하게)**

정량 목표(예시 기준이며 데이터 분석 후 조정)

- 이탈 예측 모델 ROC-AUC 0.83 이상, PR-AUC 0.63 이상
- 이탈 고객 재현율(Recall) 0.75 이상 (떠날 고객을 놓치지 않는 것이 더 중요)
- 위험 고객 1건당 맞춤 리텐션 메시지 자동 생성 100% 자동화
- 위험도 구간별 액션 자동 분기 규칙 구현

정성 목표

- API 키 없이도 전체 데모가 동작(시뮬레이션 모드)
- 핵심 로직은 테스트(TDD)로 검증

**(나) 수행 내용 (모듈 단위 · 필수)**

| 모듈 | 명칭 | 핵심 수행 내용 |
|---|---|---|
| M1 | EDA(데이터 이해) | 기본정보 탐색, 결측치 분포 확인, 변수별 정보 확인 |
| M2 | Feature Engineering | 기본 Data 전처리, 최종 데이터프레임 구성 |
| M3 | Modeling | 분할(Train/Test/Val), RandomForest + 부스팅 1종, Recall/PR-AUC 평가·임계값 |
| M4 | 리텐션 메시지 생성 | LLM으로 고객 맞춤 메시지 자동 작성(시뮬레이션 모드) |
| M5 | 리텐션 에이전트 | 위험도에 따라 액션 결정(메시지/상담연결/관찰) — 규칙 기반 |
| M6 | 통합 & 발표 | 엔드투엔드 통합, 회귀 테스트, 발표 |
| (공통) | 평가 & 테스트(TDD) | 단위 테스트, 성능 검증, 회귀 방지 |

## 2. 상세 구현/분석 방법 및 예시

### 1) 구현/분석 프로세스

각 모듈을 "문제·가치정의 → 분석(EDA) → 설계(인터페이스/테스트 정의) → TDD 개발" 순서로 진행하고 다음 모듈로 넘어갑니다. 모듈(M1~M6) 단위 순차 진행이며, 필요 시에만 앞 단계로 되돌아가 보완합니다.

전체 데이터 흐름(엔드투엔드 · 필수):

```
고객 데이터(CSV)
      │
      ▼
 [전처리]  ──▶  [예측 모델(RF + 부스팅)]  ──▶  이탈 확률(0~1)
                                              │
                                              ▼
                                    [규칙 기반 에이전트] ── 위험도 분기 ──┐
                                              │                          │
                                  위험 높음   │   위험 중간   │  위험 낮음
                                    ▼         ▼              ▼
                            [LLM 맞춤 메시지] [상담 연결]   [관찰만]
                                              │
                                              ▼
                                     [결과 로그 기록]
```

### 2) 구현/분석 방법론

**(가) 분석 방법론: CRISP-DM 매핑**

| CRISP-DM 단계 | 이 프로젝트에서 하는 일 |
|---|---|
| 비즈니스 이해 | 이탈의 비즈니스 비용 정의, 목표 지표 설정 |
| 데이터 이해 | EDA: 기본정보·결측·변수별 분포 파악 |
| 데이터 준비 | 전처리, 인코딩, 최종 데이터프레임 |
| 모델링 | RandomForest + 부스팅 1종, 가중치·임계값 튜닝 |
| 평가 | Recall/PR-AUC 중심 평가 |
| 배포(경량) | 규칙 기반 에이전트 + LLM 메시지(시뮬레이션) |

**(나) 데이터 분할·불균형 처리**

- **데이터 분리**: Train(학습) / Test(평가) / Val(제출용 홀드아웃)으로 분리하며, 이탈 비율을 유지(stratify).
- **불균형 처리**: 모델의 **클래스 가중치(class_weight / 부스팅 가중치)**로 소수 클래스(이탈)에 비중을 둡니다. (SMOTE는 이번 범위 밖)
- **임계값**: 기본 0.5가 아니라, 이탈 누락을 줄이도록 임계값을 낮춰 조정합니다.

**(다) 평가 지표 선택 근거**

이탈 고객(소수)을 놓치는 손해가 크므로 **Recall과 PR-AUC를 주지표**로, ROC-AUC는 보조로 봅니다. 모든 후보 모형을 동일한 Test 셋에서 같은 지표로 비교합니다.

**(라) 모델 구성 (필수)**

| 모형 | 역할 | 선정 이유 |
|---|---|---|
| RandomForestClassifier | 베이스라인 | 안정적 기준선, 과적합에 강건, 튜닝 부담 적음 |
| XGBoost 또는 LightGBM | 본 모형(부스팅 1종) | 불균형 보정·강력한 성능 |

> RandomForest로 기준선을 잡고 부스팅 1종으로 성능을 끌어올린 뒤, 둘 중 Test 성능이 좋은 모형을 최종 채택합니다.

**(마) LLM 프롬프트 설계 원칙**

- 입력: 고객 특징(계약유형·요금·가입기간 등) + 위험도
- 출력: 정중하고 구체적인 리텐션 메시지(혜택 제안 포함)
- 가드레일: 과장/허위 혜택 금지, 사실 기반, 톤 일관
- **시뮬레이션 모드**: API 키가 없으면 템플릿 기반 메시지로 대체하여 전체 데모가 끊기지 않게 합니다.

**(바) 에이전트 의사결정 규칙 (규칙 기반)**

| 위험도 구간 | 등급 | 액션 |
|---|---|---|
| 0.7 이상 | 고위험 | LLM 맞춤 메시지 생성 + 혜택 제안 |
| 0.4 ~ 0.7 | 중위험 | 상담사 연결 추천 |
| 0.4 미만 | 저위험 | 별도 액션 없이 관찰 |

**(사) TDD 방법**

테스트를 먼저 정의하고(Red) → 통과하는 최소 구현(Green) → 정리(Refactor)합니다. 데이터 로드·전처리·임계값 분기·에이전트 규칙처럼 **입출력이 명확한 부분**부터 테스트로 보호합니다.

### 3) 구현/분석 상세 내용 및 예시

각 모듈의 목적·입력·처리·출력을 정리합니다. (수치는 설명용 예시이며 실제 분석 후 달라질 수 있습니다.)

#### [M1] EDA (데이터 이해)

| 단계 | 내용 |
|---|---|
| **기본정보 탐색** | 데이터 크기(7,043×21), 컬럼별 자료형, 기본/부가 구성, 타깃 클래스 비율(약 26.5% 이탈) 확인 |
| **결측치 분포 확인** | 컬럼별 결측 점검 — 특히 `TotalCharges` 공백 11건(신규 고객) 식별 및 처리 방향 결정 |
| **변수별 정보 확인** | 수치형(분포·이상치), 범주형(빈도), 각 변수와 이탈률의 관계(예: 월단위 계약·짧은 tenure의 높은 이탈률) 파악 |

> 예시(목표 산출): 계약유형별·tenure 구간별 이탈률 비교 등 핵심 인사이트 도출.

#### [M2] Feature Engineering

| 단계 | 내용 |
|---|---|
| **기본 Data 전처리** | `TotalCharges` 숫자 변환·결측 대체, `Churn` 1/0 변환, 수치형 표준화, 범주형 인코딩(원-핫) |
| **최종 데이터프레임** | 기본 + 부가 Data를 결합해 모델 입력용 최종 DF 구성, `customerID` 등 식별자 제외 |

> 핵심 포인트: 전처리·인코딩은 학습 데이터로만 적합(fit)하고 검증/제출 데이터에는 적용(transform)만 하여 데이터 누수를 차단합니다.
> (파생변수 생성은 권장 확장 항목으로, 시간 여유 시 v2 문서 기준으로 추가)

#### [M3] Modeling

| 단계 | 내용 |
|---|---|
| **데이터 샘플링** | Train/Test/Val(제출용)로 분리(stratify) |
| **모델 학습** | RandomForest + 부스팅 1종 학습, 클래스 가중치로 불균형 보정 |
| **평가·임계값** | Test 셋에서 Recall·PR-AUC·ROC-AUC 비교, 임계값 튜닝 |
| **출력** | 성능 리포트, 최종 채택 모델(파일 저장) |

> 예시(목표 성능): 부스팅 계열에서 ROC-AUC 0.83+, PR-AUC 0.63+, Recall 0.75+ (임계값 조정 적용) 기대.

#### [M4] 리텐션 메시지 생성 (LLM, 시뮬레이션 모드 지원)

| 항목 | 내용 |
|---|---|
| 목적 | 고객·위험도에 맞춘 정중한 리텐션 메시지 자동 작성 |
| 입력 | 고객 특징 + 위험도 |
| 처리 | 프롬프트 구성 → LLM 호출(키 없으면 템플릿 기반 시뮬레이션으로 폴백) |
| 출력 | 맞춤 리텐션 메시지 텍스트 |
| 핵심 포인트 | API 키 없이도 전체 데모 동작, 가드레일(허위·과장 혜택 금지) 준수 |

> 예시(개념): "약정 시 월 요금 15% 할인과 기술지원 무료 혜택을 제안드립니다…" 형태의 정중한 한국어 메시지.

#### [M5] 리텐션 에이전트 (규칙 기반 의사결정)

| 항목 | 내용 |
|---|---|
| 목적 | 위험도에 따라 적절한 리텐션 액션을 자동 결정 |
| 입력 | 고객 위험도 |
| 처리 | 위험 구간 판정(고/중/저) → 액션 분기, 고위험은 메시지 생성 연계 |
| 출력 | 액션(메시지 발송 / 상담 연결 / 관찰) + (해당 시) 생성 메시지 |

> 예시(개념): 위험도 0.82 → 메시지 발송, 0.52 → 상담사 연결, 0.18 → 관찰.

#### [M6] 통합 & 발표

- 전처리 → 예측 → 에이전트 → 메시지 → 로그까지 **엔드투엔드로 연결**하고, 한 번의 실행으로 전체 흐름이 동작하도록 통합합니다.
- 결과(고객ID/위험도/액션/메시지)를 표·그래프로 정리해 발표 자료와 데모를 준비합니다.
- 전체 회귀 테스트(E2E)로 통합 후에도 핵심 흐름이 깨지지 않음을 확인합니다.

#### [공통] 평가 & 테스트(TDD)

- 입출력이 명확한 핵심 로직(데이터 로드·전처리·임계값 분기·에이전트 규칙)을 자동화된 테스트로 보호합니다.
- 성능 임계값 회귀 테스트로 모델 품질 저하를 조기에 감지합니다.

### 4) 데이터 리소스 활용 상세

**(가) 컬럼별 활용 전략**

| 활용 목적 | 사용 컬럼(그룹) | 방법 |
|---|---|---|
| 예측 입력(필수) | 기본 Data: tenure, MonthlyCharges, Contract 등 | 인코딩/스케일링 후 모델 입력 |
| 예측 입력(보조) | 부가 Data: 인터넷·부가 서비스 | 인코딩 후 입력 |
| 메시지 생성 | 계약/요금/가입기간 + 위험도 | LLM 프롬프트 입력 |

**(나) 데이터 누수(leakage) 방지 원칙**

- 전처리(인코딩·스케일링·결측대체)는 반드시 **학습 데이터로만** 수행하고, 검증/제출 데이터에는 변환만 적용합니다.
- 타깃(Churn)에서 파생된 정보를 입력 피처로 넣지 않습니다.
- `customerID`는 식별자이므로 학습 피처에서 제외합니다.

**(다) 데이터 버전 관리**

- 원본 데이터는 절대 수정하지 않고, 전처리 결과만 별도 산출물로 저장합니다.
- 전처리 규칙이 바뀌면 결과 파일명에 버전을 붙여 이력을 관리합니다.

### 5) 예상 계획 및 일정

> **일정 현실성**: 아래는 **필수 범위만** 담은 일정으로, 4인 학생팀이 8주에 **여유 있게 완수 가능**합니다. 7주차에 통합·버퍼를 두어 앞선 지연을 흡수합니다. 진도가 빠르면 v2의 권장·스트레치(파생변수·부스팅 추가·SMOTE·SHAP·대시보드 등)를 선택적으로 더합니다.

| 주차 | 단계 | 핵심 작업(필수) | 주요 산출물 |
|---|---|---|---|
| 1주 | 셋업 + EDA | 환경 구성, 데이터 적재·검증, EDA(기본정보·결측·변수별) | 계획서 확정, EDA 결과 |
| 2주 | Feature Engineering | 기본 Data 전처리, 최종 데이터프레임 | 전처리 파이프라인 |
| 3주 | 베이스라인 모델링 | 분할(Train/Test/Val), RandomForest, 평가체계(Recall/PR-AUC)·임계값 | 베이스라인 리포트 |
| 4주 | 본 모형·확정 | 부스팅 1종 학습·비교, 임계값 튜닝, 최종 모델 확정 | 모델 비교표, 최종 모델 |
| 5주 | 처방① LLM 메시지 | LLM 메시지 생성(시뮬레이션 포함), 가드레일 | 메시지 생성 모듈 |
| 6주 | 처방② 리텐션 에이전트 | 규칙 기반 에이전트(위험 분기), 액션 로깅, 통합 추론 흐름 | 에이전트 모듈 |
| 7주 | 통합 + 버퍼 | 엔드투엔드 통합, 밀린 작업 따라잡기, 결과 정리 | 통합 데모 |
| 8주 | 마무리·발표 | E2E 회귀 테스트, 발표자료, README | 데모, 발표자료 |

> 각 주차의 핵심 작업에는 해당 **핵심 로직 테스트(TDD)**를 함께 작성합니다.

**(가) 역할 분담 예시 (4인 팀 가정)**

- 데이터/EDA·전처리 담당: M1~M2
- 모델링 담당: M3
- LLM·에이전트 담당: M4~M5
- 통합/발표·문서 담당: M6 (테스트는 전원 공동)

**(나) 리스크 및 대응**

| 리스크 | 대응 |
|---|---|
| 불균형으로 성능이 낮게 보임 | Recall/PR-AUC 중심 평가, 클래스 가중치·임계값 튜닝 |
| LLM API 비용/키 부재 | 시뮬레이션 모드로 전체 데모 동작 보장 |
| 일정 지연 | 5~6주(메시지·에이전트)를 사수, 모델 튜닝(3~4주)은 부스팅 1종에서 끊기 |

| 라이브러리 호환 오류(예: Python 3.14) | 3.11/3.12 가상환경 + 버전 고정으로 팀 환경 통일 |
